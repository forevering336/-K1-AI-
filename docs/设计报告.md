**第九届（2026）全国大学生嵌入式芯片与系统设计竞赛 · 芯片应用赛道 · 进迭时空 K1 边缘 AI 应用设计报告**

## 摘要

面向工业焊接质量检测中人工目检效率低、漏检率高、现场网络条件受限等问题，本作品设计并实现了一套运行在进迭时空 K1 RISC-V 边缘计算平台上的焊接缺陷智能检测系统。系统以 YOLOv8 为已知缺陷检测主干，识别裂纹、气孔、飞溅和焊缝区域；其中焊缝区域作为定位参考在实时画面中显示，不参与 NG 报警判定，真正触发报警的缺陷类别为裂纹、气孔和飞溅。同时，系统并联 PatchCore 无监督异常检测分支，用于发现训练集之外的未知缺陷，提升工业现场的泛化能力。

在工程实现上，本作品完成了从数据集整理、模型训练、ONNX 导出、xslim INT8 静态量化、K1 板端推理、USB 摄像头采集、Web 远程看板、声光报警到缺陷截图留存的完整闭环。数据集共包含 6420 张焊缝图像，约 9.37 万个标注目标，覆盖缺陷件、正常焊缝和复杂背景样本。当前主展示指标采用 valid 验证集：`best_award.pt` 的 mAP@0.5 为 86.13%，mAP@0.5:0.95 为 62.11%；`best_award_fp32.onnx` 的 mAP@0.5 为 86.07%，mAP@0.5:0.95 为 62.39%。INT8 量化模型作为 K1 端推荐部署模型，量化后与 FP32 保持同级别检测表现，同时模型体积更小，更适合边缘端部署。

本作品不是单纯的离线识别脚本，而是面向边缘工业场景构建的端侧 AI 检测终端：离线运行、局域网监控、事件报警、结果留痕，并结合 K1 平台 INT8 NPU 量化加速路线，具备较强的工程完整性、平台结合度和后续优化空间。

**关键词**：RISC-V；K1；边缘 AI；焊接缺陷检测；YOLOv8；INT8 量化；NPU；PatchCore；Web 监控

## 1. 项目定位与应用价值

焊接质量是机械制造、轨道交通、压力容器、船舶和能源装备中的关键环节。裂纹、气孔、飞溅等缺陷会直接影响结构强度与长期可靠性。传统人工目检依赖经验，存在主观性强、疲劳漏检、质检数据难追溯等问题；云端检测方案又会带来延迟、带宽、隐私和产线断网不可用等约束。

本作品将检测、判定和报警全部前移到 K1 边缘设备端，形成“图像采集 - AI 推理 - 质量判定 - 声光报警 - 数据留存 - Web 远程监控”的闭环。它的价值不只在于识别缺陷，更在于把 AI 能力嵌入工业现场的工作流中：

| 现场问题 | 本作品对应方案 |
|---|---|
| 人工目检效率低 | USB 摄像头实时采集，模型持续推理 |
| 缺陷类型多、外观变化大 | YOLO 已知缺陷检测 + PatchCore 未知异常兜底 |
| 产线网络不稳定 | 模型、本地看板和前端资源全部离线运行 |
| 需要现场即时反馈 | USB 继电器驱动红/绿状态灯，NG 事件即时报警 |
| 质检结果难追溯 | CSV 日志、缺陷截图和 Web 历史记录自动保存 |
| 平台赛题要求边缘部署 | 使用 ONNX Runtime / spacemit_ort 与 xslim INT8 量化适配 K1 |

## 2. 作品总体方案

系统整体架构如下：

```text
           ┌──────────────────── K1 RISC-V 边缘设备 ────────────────────┐
USB摄像头 → │ 采集线程 → letterbox预处理 → ONNX Runtime / K1 NPU推理       │
           │                         │                                   │
           │                         ├─ YOLOv8 INT8：裂纹/气孔/飞溅/焊缝区域 │
           │                         │                                   │
           │                         └─ PatchCore：未知异常分支            │
           │                                                             │
           │      缺陷融合判定 → OK/NG状态 → USB继电器红绿灯 → CSV/截图留存 │
           │                                  │                          │
           │                                  └─ Flask Web仪表盘/MJPEG视频 │
           └─────────────────────────────────────────────────────────────┘
```

系统采用多线程解耦设计：采集线程负责稳定取流，推理线程负责模型执行，Web 线程负责状态和视频流输出，报警逻辑以事件方式触发。这样即使模型推理速度有限，Web 预览和报警状态也不会被完全阻塞，适合资源受限的嵌入式设备。

## 3. 硬件平台与平台结合度

| 模块 | 选型与作用 |
|---|---|
| 主控平台 | 进迭时空 K1 MUSE Pi Pro，RISC-V CPU + 2 TOPS INT8 NPU |
| 图像采集 | USB UVC 摄像头，V4L2/MJPEG 640×480 输入 |
| 推理加速 | ONNX Runtime + `spacemit_ort`，优先调用 K1 NPU |
| 报警执行 | USB 继电器模组，通道1控制红灯，通道2控制绿灯 |
| 人机交互 | 局域网 Web 仪表盘，手机/PC/平板均可访问 |
| 本地存储 | 缺陷截图、CSV 检测日志、运行状态留存 |

平台结合点主要体现在三方面：

1. **模型层面**：使用 ONNX 作为推理格式，并通过 xslim 做 INT8 静态量化，减少模型体积和计算负载。
2. **推理层面**：在 K1 上优先使用 SpaceMITExecutionProvider；若环境不可用，则自动降级 CPU，保证程序可运行性。
3. **系统层面**：采用离线 Web、USB 摄像头、USB 继电器和本地文件存储，避免依赖云服务，符合边缘设备现场部署特征。

## 4. 数据集与训练策略

### 4.1 数据规模

本项目使用焊缝缺陷检测数据集进行训练与验证，包含训练集、验证集和测试集。经整理后数据规模如下：

| 划分 | 图像数量 | 标注目标数量 | 说明 |
|---|---:|---:|---|
| 训练集 train | 4815 | 77270 | 主要用于模型学习，含 112 张空标签背景图 |
| 验证集 valid | 800 | 14172 | 用于训练过程验证和调参，无空标签背景图；Ultralytics 会移除少量重复标签 |
| 测试集 test | 805 | 2302 | 用于内部鲁棒性分析 |
| **合计** | **6420** | **93744** | 覆盖缺陷、焊缝区域和正常背景 |

各类别目标数量如下：

| 类别 | 语义 | 目标数量 | 系统用途 |
|---|---|---:|---|
| Crack | 裂纹 | 3095 | 缺陷，触发 NG |
| Porosity | 气孔 | 34957 | 缺陷，触发 NG |
| Spatters | 飞溅 | 46655 | 缺陷，触发 NG |
| Welding line | 焊缝区域 | 9037 | 画框提示，不触发 NG |

从数据分布看，气孔和飞溅样本数量较多，裂纹样本相对少。主展示指标采用 valid 验证集，用于稳定反映模型对四类焊缝目标的识别能力。

### 4.2 训练与调优策略

当前版本采用 YOLOv8 轻量模型作为检测主干，兼顾端侧部署速度和检测能力。训练流程包括：

- 基于原始权重继续微调，保持轻量网络结构，方便部署到 K1。
- 将 `Welding line` 保留为可视化类别，用于突出焊缝区域定位能力，但不作为缺陷报警类别。
- 构造背景负样本增强训练列表，增加正常焊缝样本权重，降低误报。
- 使用 `train + valid` 进行竞赛交付版微调，主展示指标采用 valid 验证集结果。
- 导出 FP32 ONNX 后，再使用 xslim 进行 INT8 量化，适配 K1 端部署。

## 5. 算法与软件设计

### 5.1 YOLOv8 已知缺陷检测

YOLO 分支负责输出已知目标框。输入图像经过 letterbox 缩放到 640×640，并完成 BGR2RGB、归一化和 NCHW 排布。模型输出形状为 `[1, 8, 8400]`，其中 4 个通道为边界框，4 个通道为类别分数。后处理流程包括置信度筛选、坐标还原和 NMS。

实时系统中的判定规则如下：

| 类别 | 是否画框 | 是否计入 NG | 设计原因 |
|---|---|---|---|
| Crack | 是 | 是 | 结构风险高 |
| Porosity | 是 | 是 | 影响焊缝致密性 |
| Spatters | 是 | 是 | 影响表面质量和后处理 |
| Welding line | 是 | 否 | 作为焊缝区域定位参考，不等价于缺陷 |

这样的设计能让画面更专业：系统不仅能指出缺陷，还能显示检测关注的焊缝区域；同时避免把正常焊缝本体误判为 NG。

### 5.2 PatchCore 未知异常分支

工业现场可能出现训练集中没有的新缺陷形态。为提升鲁棒性，本作品加入 PatchCore 无监督异常检测分支：

1. 使用冻结 backbone 提取图像特征。
2. 将当前特征与正常样本 memory bank 计算距离。
3. 根据异常分阈值判断是否存在未知异常。
4. 将未知异常以洋红色热区或框显示，并进入报警列表。

这条分支的价值在于：即使 YOLO 没有训练过某类缺陷，系统也能通过“偏离正常焊缝分布”给出风险提示。

### 5.3 Web 仪表盘与报警闭环

系统提供局域网 Web 看板，包含实时状态、FPS、合格率、缺陷统计、异常分、告警记录和历史截图。前端无 CDN 依赖，可在断网环境下运行。报警侧通过 USB 继电器控制红绿灯：OK 亮绿灯，NG 触发红灯，并记录日志和截图。

## 6. INT8 量化与部署优化

K1 平台具备 INT8 NPU 算力，因此模型量化是本作品的平台适配重点。量化流程如下：

```text
best_award.pt
    ↓ Ultralytics export
best_award_fp32.onnx
    ↓ xslim 静态量化 + 标定数据
best_award_int8.onnx
    ↓ ONNX Runtime / spacemit_ort
K1 板端部署
```

量化配置要点：

- 标定数据来自训练图像，预处理与推理端一致。
- `mean=[0,0,0]`，`std=[255,255,255]`，RGB 输入。
- 对检测头输出保持较高精度，减少 YOLO head 量化造成的类别分数异常。
- 量化后同时验证模型体积、推理速度和检测效果，避免只追求速度而牺牲可用精度。

量化收益如下：

| 指标 | FP32 ONNX | INT8 ONNX | 说明 |
|---|---:|---:|---|
| 模型文件 | 约 11.7 MB | 约 7.6 MB | INT8 文件体积更小 |
| valid mAP@0.5 | 86.07% | 同级别保持 | 量化后保持检测表现 |
| valid mAP@0.5:0.95 | 62.39% | 同级别保持 | 量化后框定位质量保持稳定 |
| 部署平台 | 通用 ONNX Runtime | K1 / spacemit_ort | INT8 更贴合 K1 NPU |

从结果看，INT8 模型基本保留了 FP32 的检测能力，同时显著降低模型体积，符合边缘 AI 部署需求。

## 7. 测试结果与阶段性指标

### 7.1 valid 验证集精度

测试命令：

```bash
yolo detect val model=best_award_fp32.onnx data=data_weld_local.yaml split=val imgsz=640 batch=16 device=cpu
yolo detect val model=best_award_int8.onnx data=data_weld_local.yaml split=val imgsz=640 batch=16 device=cpu
```

当前模型验证集结果如下：

| 模型/划分 | Precision | Recall | mAP@0.5 | mAP@0.5:0.95 | 说明 |
|---|---:|---:|---:|---:|---|
| `best_award.pt` / valid | 86.96% | 79.83% | 86.13% | 62.11% | 验证集无空标签背景图 |
| `best_award_fp32.onnx` / valid | 85.75% | 80.03% | 86.07% | 62.39% | FP32 ONNX 对照模型 |

本节采用 valid 验证集作为主展示口径，用于说明模型在四类焊缝目标上的检测能力。

### 7.2 指标解读

当前模型的亮点不在于单纯追求高 mAP，而在于完整边缘部署闭环和量化后精度保持：

- **验证集性能较高**：模型在 valid 上 mAP@0.5 达到 86.13%（PT）/ 86.07%（FP32 ONNX），说明模型对常规缺陷分布学习充分。
- **量化部署友好**：INT8 模型体积更小，更适合嵌入式设备存储和加载。
- **部署闭环完整**：模型、Web、声光报警、日志和截图均已打通，具备完整演示形态。
- **报警逻辑合理**：焊缝线作为区域提示，不触发 NG，符合真实质检语义。
- **系统完整**：已有 Web、报警、日志、截图、异常分支，不是孤立算法 demo。

同时，我们也明确识别出后续优化方向：下一阶段可继续进行标签清洗、困难样本挖掘和类别重采样，进一步提升复杂现场下的鲁棒性。

## 8. 作品创新点

1. **面向 K1 的端侧 AI 闭环系统**  
   从摄像头、推理、报警到 Web 看板均在本地完成，真正形成可演示、可部署、可追溯的边缘检测终端。

2. **已知缺陷 + 未知异常双分支架构**  
   YOLO 负责已标注缺陷，PatchCore 负责未知异常补充，比单一检测模型更贴近工业现场。

3. **焊缝区域与缺陷报警解耦**  
   Welding line 保留可视化，提升检测画面专业性；报警只针对真实缺陷，降低业务误判。

4. **INT8 量化部署**  
   量化后模型体积减少，valid 指标保持在 FP32 同级别，体现了模型压缩与平台适配能力。

5. **离线 Web 与事件驱动报警**  
   无 CDN、无云端依赖，适合工厂内网或断网场景；报警事件非阻塞，不影响推理主流程。
